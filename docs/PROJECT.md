# Pokefolio 프로젝트 정의

## 개요

Pokefolio는 Pokémon TCG 수집가가 보유 카드와 팩 개봉 이력을 기록하고, 자신의 컬렉션을 한눈에 이해할 수 있도록 돕는 웹 애플리케이션입니다.

## 제품 목표

- 모바일 환경에서 카드를 빠르게 조회하고 등록할 수 있게 합니다.
- 컬렉션과 개봉 기록을 일관된 데이터로 관리합니다.
- 바인더 구성, 교환, 컬렉션 분석을 위한 기반 데이터를 제공합니다.
- 작은 규모로 시작하되 기능 확장이 가능한 경계를 유지합니다.

## 초기 범위

초기 제품은 다음 영역을 대상으로 합니다.

1. 카드 컬렉션 등록, 조회, 수정, 삭제
2. 팩 개봉과 획득 카드 기록
3. 카드 및 컬렉션 검색과 필터링
4. 모바일 중심의 컬렉션 요약 화면

AI 바인더 배치, 교환 추천, 고급 인사이트는 핵심 데이터와 사용자 흐름이 안정된 뒤 확장합니다.

## 기술 구성

### 프런트엔드

- Next.js와 App Router를 사용합니다.
- Mobile First로 설계하고 큰 화면으로 점진 확장합니다.
- 백엔드 API를 유일한 서버 데이터 접근 경계로 사용합니다.

### 백엔드

- Python과 FastAPI를 사용합니다.
- API, 서비스, 저장소, 데이터 모델의 책임을 분리합니다.
- 요청과 응답 데이터는 명시적인 스키마로 검증합니다.

### 데이터베이스

- 초기 데이터베이스는 SQLite를 사용합니다.
- 데이터베이스 파일은 런타임 산출물로 취급하며 Git에 포함하지 않습니다.
- 데이터 접근 계층을 분리해 향후 다른 데이터베이스로 전환할 여지를 둡니다.

### 저장소

- 프런트엔드, 백엔드, 공용 문서를 하나의 모노레포에서 관리합니다.
- 각 애플리케이션은 독립적인 의존성과 실행 명령을 가집니다.
- API 계약은 문서와 테스트를 통해 양쪽에서 공유합니다.

## 아키텍처 경계

```text
Mobile-first Next.js UI
          │
          │ HTTP/JSON
          ▼
      FastAPI API
          │
    Service layer
          │
   Repository layer
          │
        SQLite
```

프런트엔드는 데이터베이스에 직접 접근하지 않습니다. API는 입력과 HTTP 처리를 담당하고, 서비스는 비즈니스 규칙을, 저장소는 영속성 처리를 담당합니다.

## 비기능 요구사항

- 모바일 화면에서 핵심 작업을 우선 제공할 것
- 키보드와 보조 기술을 고려한 접근성을 확보할 것
- 사용자 입력을 API 경계에서 검증할 것
- 핵심 비즈니스 로직을 자동화 테스트로 보호할 것
- 비밀 정보와 사용자 데이터를 저장소에 커밋하지 않을 것

## 현재 결정하지 않은 사항

- 인증 및 사용자 계정 방식
- 카드 데이터 외부 제공자
- 배포 및 호스팅 플랫폼
- 프런트엔드 스타일링 도구
- ORM과 마이그레이션 도구
- AI 기능의 모델과 제공자

이 항목들은 해당 개발 단계에서 요구사항과 운영 비용을 검토한 뒤 결정합니다.
