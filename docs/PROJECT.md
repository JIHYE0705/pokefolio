# Pokefolio 프로젝트 정의

## 제품 비전

Pokefolio는 포켓몬 카드 컬렉터가 자신의 카드를 편안하게 모으고, 기록하고, 꾸미고, 교환할 수 있게 돕는 모바일 우선 웹 서비스다. 가격 변동이나 투자 수익보다 개인 컬렉션의 기억과 취향, 수집의 성취감을 중심에 둔다.

## 사용자 문제

- 보유 카드와 수량이 여러 장소에 흩어져 전체 컬렉션을 파악하기 어렵다.
- 중복 카드와 위시리스트를 따로 관리해 교환 준비가 번거롭다.
- 실제 바인더처럼 카드를 배치하고 감상할 디지털 공간이 부족하다.
- 팩을 언제 열었고 어떤 카드를 얻었는지 시간이 지나면 기억하기 어렵다.
- 기존 서비스는 시세·거래 중심이어서 개인적인 수집 경험이 뒤로 밀리기 쉽다.

## 주요 사용자

1. **기록형 컬렉터**: 보유 카드, 수량, 상태와 메모를 정확히 남기고 싶다.
2. **꾸미기형 컬렉터**: 카드 이미지가 돋보이는 바인더를 구성하고 감상하고 싶다.
3. **개봉 기록형 컬렉터**: 팩을 연 날짜, 세트, 획득 카드와 느낌을 일기로 보관하고 싶다.
4. **교환 준비형 컬렉터**: 위시리스트와 중복 카드를 비교해 교환 후보를 찾고 싶다.

초기 MVP는 로그인 없는 단일 사용자를 전제로 한다.

## 핵심 가치

- **소유의 명확함**: 내가 가진 카드와 수량을 빠르게 확인한다.
- **수집의 즐거움**: 카드 이미지와 바인더가 중심이 되는 따뜻한 앨범 경험을 제공한다.
- **기록의 지속성**: 개봉의 순간과 개인 메모를 다시 볼 수 있다.
- **교환의 준비**: 위시리스트와 중복 정보를 구조화한다.
- **보조적 추천**: AI나 자동화는 사용자의 판단을 대신하지 않고 정리와 발견을 돕는다.

## 핵심 사용자 흐름

### 카드 수집

```text
카드 검색 → 카드 상세 확인 → 내 컬렉션에 추가 → 수량·상태·메모 확인
```

### 바인더 꾸미기

```text
바인더 생성 → 페이지 확인 → 보유 카드 선택 → 슬롯에 배치 → 바인더 감상
```

### 개봉일기 기록

```text
새 기록 시작 → 날짜·세트·메모 입력 → 획득 카드 추가 → 저장 → 기록 상세 확인
```

### 위시리스트와 교환 준비

```text
카드 상세 → 위시리스트 추가 → 중복 카드 확인 → 향후 교환 후보 비교
```

## MVP 포함 범위

- 카드 카탈로그 조회와 검색
- 보유 카드 추가, 수량·상태·메모 관리
- 위시리스트 추가와 제거
- 바인더 생성, 페이지와 슬롯 단위 카드 배치
- 개봉일기 생성, 목록, 상세와 획득 카드 기록
- 홈에서 컬렉션과 최근 기록 요약
- loading, empty, error, success 상태와 기본 접근성
- 로컬 SQLite에 저장되는 로그인 없는 단일 사용자 데이터

## MVP 제외 범위

- 로그인, 계정, 다중 사용자, 권한 관리
- 카드 시세, 투자 수익률, 결제와 마켓플레이스
- 실시간 교환 채팅, 배송, 거래 중재
- AI 자동 바인더 구성이나 생성형 콘텐츠
- OCR·카메라 기반 카드 인식
- 푸시 알림, 오프라인 PWA 설치 경험
- Docker 기반 로컬 개발과 운영 배포 자동화

## 향후 확장

- 보유·위시·바인더 패턴을 활용한 취향 분석
- 설명 가능한 규칙 기반 카드 및 교환 추천
- OCR과 이미지 인식을 이용한 등록 보조
- 사용자 간 교환 후보 매칭과 안전한 연락 흐름
- 계정, 동기화, 백업과 다중 기기 사용
- PostgreSQL 전환과 운영 환경 구성
- PWA 설치, 캐시, 제한적 오프라인 조회

## 성공 기준

MVP 사용자 테스트에서 다음을 확인한다.

- 사용자가 도움 없이 카드 검색부터 보유 카드 추가까지 완료한다.
- 사용자가 보유 수량과 중복 여부를 한 화면 흐름에서 확인한다.
- 사용자가 3×3 바인더 페이지에 카드를 배치하고 다시 찾는다.
- 사용자가 개봉일기를 생성하고 획득 카드를 연결한다.
- 주요 흐름이 390px에서 가로 스크롤 없이 동작한다.
- 핵심 작업이 키보드와 보조 기술로 가능하고 모든 데이터 화면에 복구 가능한 상태가 있다.
- 사용자 인터뷰에서 서비스가 가격 도구보다 개인 수집 앨범으로 인식된다.

정량 목표값은 실제 MVP 사용자 모집 규모와 분석 도구를 정한 뒤 확정한다.

## 기술 경계

```text
Next.js mobile-first UI
          │ HTTP/JSON
          ▼
      FastAPI router
          ▼
       Service
          ▼
      Repository
          ▼
SQLAlchemy 2.x + SQLite
```

- DB 모델은 SQLAlchemy 2.x typed declarative mapping으로 작성한다.
- API 요청·응답은 별도 Pydantic v2 스키마로 검증한다.
- 스키마 변경은 Alembic으로 관리한다.
- SQLModel은 사용하지 않는다.
- native SQL과 SQLite 전용 기능을 최소화해 PostgreSQL 이전 가능성을 유지한다.

## 미결정 사항

- 카드 메타데이터 외부 제공자와 라이선스
- 카드 식별자의 내부·외부 키 전략
- 카드 이미지 저장·캐시 정책
- 페이지네이션 방식과 기본 크기
- 배포 플랫폼과 운영 데이터베이스 전환 시점
- 로그인 도입 시 사용자 데이터 이전 방식
