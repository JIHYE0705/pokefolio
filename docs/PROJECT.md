# Pokefolio 프로젝트 정의

Pokefolio의 최상위 제품 판단 기준은 [제품 비전](PRODUCT_VISION.md)에 있다.

## 제품 비전

Pokefolio는 사용자가 포켓몬 카드를 모으며 느낀 설렘, 뿌듯함, 추억을 기록하고 언제든 다시 꺼내볼 수 있게 하는 모바일 컬렉션 저널이다.

> 언제든 내가 모은 카드와, 그 카드를 모았던 순간까지 다시 꺼내볼 수 있는 공간.

카드 데이터베이스, 시세와 보유 수량은 이 경험을 돕는 정보이지 제품의 중심이 아니다.

## 핵심 사용자 여정

### 수집 경험 남기기

```text
기록 유형 선택
→ 사진 또는 카드 선택(건너뛰기 가능)
→ 기억 남기기
```

빠른 기록은 사용자가 내려야 할 결정을 셋으로 제한한다. 기록 유형만 필수이고 날짜·시간은 현재 시각으로 자동 설정한다. 카드, 획득 카드 중 고르는 오늘의 카드, 한 줄 기록과 사진은 선택이다. 팩·박스 개봉, 낱장 구매, 카드샵·팝업스토어 방문, 선물, 교환, 여행과 개인적인 기억을 카드 없이도 `Moment(오늘의 순간)`로 남길 수 있어야 한다.

첫 기록이 없을 때는 결핍을 알리는 대신 `첫 번째 추억을 만들어보세요.`와 `첫 순간 기록하기`를 보여준다. 저장 후에는 toast로 끝내지 않고 날짜와 선택적인 한 줄을 담은 `오늘의 순간이 기록됐어요.` 보상 화면을 보여준다. 상세 흐름은 [Journal 빠른 기록 UX 설계](superpowers/specs/2026-07-17-journal-quick-entry-ux-design.md)를 따른다.

### 컬렉션 감상하기

```text
최근 추가 카드 확인
→ 카드 이미지와 획득한 순간 보기
→ 포켓몬·테마·확장팩·색감별 수집 취향 탐색
→ 보유 수량과 중복은 필요할 때 확인
```

### 바인더로 취향 표현하기

```text
테마가 있는 Binder 생성
→ 이름·짧은 설명·대표 카드 설정
→ 보유 카드 배치
→ 이어서 꾸미고 완성한 이야기 감상
```

### 기억과 변화 다시 보기

```text
Journal Timeline 또는 Home 진입
→ 오늘 기록한 순간·오늘의 카드·1년 전 오늘 보기
→ 이어서 꾸밀 Binder 확인
→ Collection Journey에서 취향과 바인더의 변화 발견
→ Keeper의 짧은 관찰 확인
```

### 앞으로의 수집 준비하기

```text
Wishlist에 카드 추가
→ 중복 카드 확인
→ Keeper가 발견한 연결을 참고
→ 사용자가 직접 다음 행동 결정
```

## MVP 목표

로그인 없는 단일 사용자가 카드와 수집 경험을 부담 없이 연결해 기록하고, 다시 앱을 열었을 때 최근 카드·이어 꾸밀 바인더·과거 기억 중 하나를 자연스럽게 다시 만나는 경험을 검증한다.

MVP는 완전한 카드 관리 도구보다 다음 세 가지를 먼저 증명한다.

1. Moment 기록이 데이터 입력이 아니라 기억을 남기는 경험으로 느껴진다.
2. 카드 이미지와 짧은 기록만으로도 다시 보고 싶은 개인 컬렉션이 만들어진다.
3. Home과 Binder가 사용자가 수집을 이어가도록 부담 없는 다음 행동을 제공한다.

## MVP 기능

- 카드 카탈로그 조회와 검색
- 보유 카드 추가, 수량·상태·메모와 중복 확인
- 기록 유형만 필수이고 날짜·시간은 현재 시각으로 자동 저장되는 Moment 빠른 기록
- 선택적인 획득 카드, 획득 카드 중 오늘의 카드, 한 줄 기록과 사진
- 첫 기록 Empty State, 날짜순 Journal Timeline과 저장 보상 화면
- 테마·이름·짧은 설명·대표 카드를 가진 Binder와 3×3 카드 배치
- Home의 오늘 기록한 순간, 오늘의 카드, 1년 전 오늘과 이어서 꾸밀 Binder
- 휴지통 이동, 즉시 되돌리기와 복원 가능한 기록 제거
- Wishlist 추가와 제거
- loading, empty, error, success 상태와 기본 접근성
- 로컬 SQLite에 저장되는 로그인 없는 단일 사용자 데이터

장소, 기분, 함께한 사람, 구매 정보와 Keeper의 구체적인 MVP 포함 범위는 데이터·개인정보·사용성 검토 뒤 확정한다. 현재는 구현된 기능이 아니라 제품 요구 후보이다.

## MVP에서 검증할 가설

- 사용자는 카드만 등록하는 흐름보다 Moment를 먼저 남기는 흐름에서 더 큰 애착을 느낀다.
- 기록 유형만 확인하고 나머지 항목을 건너뛸 수 있으면 Collection Journal 작성을 부담스럽게 느끼지 않는다.
- Home에서 총량 통계보다 오늘 기록한 순간, 오늘의 카드, 1년 전 오늘과 이어 꾸밀 Binder를 먼저 보여줄 때 재방문 이유가 선명해진다.
- 사용자는 테마와 이야기가 있는 Binder를 단순 슬롯 목록보다 자신의 취향을 표현하는 공간으로 인식한다.
- 과거 기록을 다시 보여주는 경험은 수집량이나 연속 기록을 강조하지 않아도 뿌듯함을 만든다.
- Keeper는 짧고 근거가 보이는 관찰만 제공할 때 방해가 아니라 동반자로 받아들여진다.

## 성공 기준

MVP 사용자 테스트에서 다음을 확인한다.

- 사용자가 도움 없이 3결정 흐름을 이해하고 기록 유형 외 선택 항목 없이도 Moment를 저장한다.
- 사용자가 필요할 때 획득 카드와 그중 오늘의 카드, 한 줄 기록과 사진을 추가한다.
- 사용자가 원치 않는 선택 항목을 건너뛰고도 기록을 저장한다.
- 사용자가 날짜순 Journal Timeline에서 카드가 있거나 없는 Moment를 다시 찾는다.
- 사용자가 테마가 있는 3×3 Binder를 만들고 이어서 꾸민다.
- 사용자가 Home에서 통계가 아니라 최근 기억 또는 이어서 할 행동을 먼저 인식한다.
- 사용자가 기록을 저장한 뒤 보상 화면에서 방금 남긴 날짜와 한 줄을 확인한다.
- 사용자가 Moment를 휴지통으로 이동한 뒤 되돌리거나 복원할 수 있음을 이해한다.
- 사용자가 과거 기록을 다시 보고 당시의 장소·기분·이야기 중 하나를 떠올린다.
- 사용자 인터뷰에서 Pokefolio가 카드 DB나 투자 도구보다 “내 수집 기억을 보관하는 공간”으로 설명된다.
- 주요 흐름이 390px에서 가로 스크롤 없이 동작한다.
- 핵심 작업이 키보드와 보조 기술로 가능하고 모든 데이터 화면에 복구 가능한 상태가 있다.

정량 목표는 MVP 사용자 모집 규모와 분석 방법을 정한 뒤 확정한다. 연속 기록, 카드 수와 구매액을 참여 압박 지표로 사용하지 않는다.

## MVP 제외 범위

- 로그인, 계정, 다중 사용자, 권한 관리와 동기화
- 카드 시세, 투자 수익률, 포트폴리오와 구매 권유
- 결제, 마켓플레이스, 실시간 교환 채팅, 배송과 거래 중재
- 생성형 AI가 필수인 기록·추천 흐름
- OCR·카메라 기반 카드 인식
- 고급 취향 점수, 경쟁 순위, streak와 성과 배지
- Moment를 평가하는 별점
- 푸시 알림, 오프라인 PWA 설치 경험
- Docker 기반 로컬 개발과 운영 배포 자동화

## 기술 경계

```text
Next.js mobile-first UI
          │ HTTP/JSON
          ▼
      FastAPI router
          ▼
       Service
          ▼
      Repository
          ▼
SQLAlchemy 2.x + SQLite
```

- DB 모델은 SQLAlchemy 2.x typed declarative mapping으로 작성한다.
- API 요청·응답은 별도 Pydantic v2 스키마로 검증한다.
- 스키마 변경은 Alembic으로 관리한다.
- SQLModel은 사용하지 않는다.
- native SQL과 SQLite 전용 기능을 최소화해 PostgreSQL 이전 가능성을 유지한다.
- Frontend와 Backend는 공개된 HTTP/JSON 계약으로만 통신한다.

- 카드 메타데이터 외부 제공자, 라이선스와 이미지 캐시 정책
- 카드 식별자의 내부·외부 키와 페이지네이션 전략
- 로그인 도입 시 사용자 데이터 이전 방식
