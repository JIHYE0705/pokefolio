# Pokefolio 디자인 원칙

이 문서는 구현 전 디자인 방향이다. 색상 값과 치수 토큰은 검증을 위한 **provisional(임시) 후보**이며 확정값이 아니다.

## 브랜드 및 UI 원칙

- 카드 이미지와 사용자의 컬렉션이 브랜드 장식보다 먼저 보인다.
- 분위기는 깔끔하고 귀여우며 따뜻한 개인 수집 앨범에 가깝다.
- 흰색 또는 밝은 중성 표면에 절제된 보라색을 행동과 선택의 강조로 사용한다.
- 익숙한 모바일 패턴을 사용하되 공식 포켓몬 앱, 로고, 아이콘, trade dress를 모방하지 않는다.
- 한 화면의 목적과 기본 행동이 설명 없이 드러나도록 한다.

## Apple HIG 적용 원칙

웹 구현에서는 Apple 고유 컴포넌트를 복제하지 않고 다음 보편 원칙을 적용한다.

- **Clarity:** 읽기 순서, 제목, 콘텐츠, 행동의 위계를 분명히 한다.
- **Deference:** 카드 이미지와 사용자 데이터가 UI chrome보다 돋보이게 한다.
- **Depth:** 그룹과 상호작용을 설명할 때만 경계·간격·약한 그림자를 사용한다.
- **Familiarity:** 하단 내비게이션, 검색, 입력, 피드백은 플랫폼에 익숙한 동작을 따른다.
- **Adaptability:** safe area, 화면 폭, 방향, 확대 텍스트, 로케일 변화에 대응한다.
- **Accessibility:** 시각·터치·키보드·스크린리더 어느 하나에만 의존하지 않는다.

## 컬러 역할

| provisional token | 역할 | 임시 예시 | 확정 조건 |
| --- | --- | --- | --- |
| `surface-canvas` | 전체 배경 | `#F7F7FA` | 실기기와 밝은 환경에서 카드 경계 확인 |
| `surface-card` | 카드·패널 표면 | `#FFFFFF` | 배경과 충분히 구분되는지 확인 |
| `text-primary` | 주요 텍스트 | `#1C1C1E` | body text WCAG AA 대비 확인 |
| `text-secondary` | 보조 텍스트 | `#66666F` | 작은 텍스트 4.5:1 이상 확인 |
| `accent-primary` | 기본 행동·선택·포커스 | `#6F5BD3` | 흰색 텍스트와 모든 상태 대비 확인 |
| `border-subtle` | 구분선·입력 경계 | `#E4E4EA` | 고대비 모드와 화면 밝기 확인 |
| `feedback-danger` | 오류·파괴 행동 | `#C93434` | 색상 외 아이콘·문구와 함께 검증 |
| `feedback-success` | 완료 상태 | `#287A4B` | 색상 외 문구·아이콘과 함께 검증 |

- 같은 색을 상호작용과 장식에 동시에 사용하지 않는다.
- 상태는 색상만으로 전달하지 않고 아이콘, 텍스트, 모양을 함께 사용한다.
- 다크 모드는 MVP 이후 검토하되 semantic token 구조로 확장 가능하게 한다.

## 타이포그래피 원칙

- 플랫폼 기본 sans-serif font stack을 우선해 다운로드와 가독성 부담을 줄인다.
- body 기본 후보는 16~17 CSS px이며 실제 브라우저 확대와 접근성 검사 후 확정한다.
- 제목은 크기와 semibold 이상의 무게로 위계를 만들고 가벼운 font weight를 피한다.
- line-height와 문단 폭을 충분히 확보하고 핵심 정보를 강제로 한 줄 말줄임하지 않는다.
- 200% 확대에서도 텍스트, 아이콘, 행동이 잘리거나 겹치지 않게 수직 적층을 허용한다.

## Spacing, radius, shadow

| 항목 | provisional 후보 | 사용 원칙 |
| --- | --- | --- |
| spacing | 4px 기반 `4/8/12/16/24/32` | 화면 바깥 여백 16px 후보, 관련 항목은 가깝게 |
| radius | `8/12/16/24px` | 같은 역할은 같은 radius, 중첩 radius 남용 금지 |
| shadow | `0 2px 10px rgb(0 0 0 / 0.06)` | 표면 구분이 border로 부족할 때만 사용 |

수치는 목업과 실기기 검증 전 임시이며 token 역할을 먼저 고정한다.

## 카드 이미지

- 원본 카드 비율 후보는 약 `2.5:3.5`이며 CSS `aspect-ratio: 5 / 7`로 검증한다.
- 이미지는 찌그러뜨리지 않고 `object-fit: contain`을 기본으로 한다.
- 카드 이름과 세트 정보를 대체 텍스트에 포함하되 장식용 중복 이미지는 빈 대체 텍스트를 사용한다.
- 버튼, 배지, 수량 표시는 카드의 이름·일러스트·세트 번호를 가리지 않는다.
- grid에서는 카드가 가장 큰 시각 요소가 되며 메타데이터는 아래에 간결하게 둔다.

## Bottom navigation

- 주요 목적지는 Home, Collection, Binder, Opening Log의 4개로 시작한다.
- Wishlist는 Home 또는 Collection에서 진입하며 사용성 검증 후 탭 승격을 검토한다.
- 현재 위치는 색상뿐 아니라 아이콘 형태·텍스트·선택 상태로 전달한다.
- 각 항목의 전체 터치 영역은 최소 44×44 CSS px를 목표로 한다.
- 하단 고정 영역은 `env(safe-area-inset-bottom)`을 포함하고 콘텐츠를 가리지 않는다.

## 컴포넌트 원칙

### 버튼

- 한 화면의 primary action은 원칙적으로 하나다.
- primary, secondary, destructive를 시각과 문구로 구분한다.
- disabled 상태는 opacity만 낮추지 않고 비활성 스타일과 semantic `disabled`를 사용한다.
- 아이콘 단독 버튼은 accessible name과 44×44 CSS px 터치 영역을 갖는다.

### 입력

- placeholder 대신 항상 보이는 label로 의미를 설명한다.
- 필수 여부, 형식, 오류를 입력 가까이에 표시하고 프로그램적으로 연결한다.
- 가능한 값은 자유 텍스트보다 선택 컨트롤을 우선한다.
- 모바일 키보드가 submit과 오류 메시지를 가리지 않게 한다.

### 검색

- Collection 상단의 한 위치에서 카드 이름, 세트, 번호를 검색한다.
- 검색 범위가 드러나는 label 또는 placeholder를 사용한다.
- 결과 없음은 오류와 구분하고 검색어 수정·필터 초기화를 제안한다.
- 검색 기록을 도입할 때는 개인정보와 삭제 동작을 먼저 정한다.

### 카드

- 이미지, 카드 이름, 세트·번호, 보유 수량 순으로 정보를 제공한다.
- 카드 전체가 링크이면 내부에 충돌하는 중첩 버튼을 두지 않는다.
- selected 상태는 outline/check 아이콘과 접근성 상태로 함께 표현한다.

### 바인더 슬롯

- 3×3 grid를 기본 후보로 하되 390px에서 카드 이미지와 터치 행동을 함께 검증한다.
- 빈 슬롯은 단순한 `+`와 "카드 추가" accessible name을 제공한다.
- 채워진 슬롯은 카드 상세 보기와 교체/비우기 행동을 명확히 분리한다.
- 드래그는 보조 입력이며 탭·키보드로 동일한 배치를 할 수 있어야 한다.

## 화면 상태

| 상태 | 원칙 |
| --- | --- |
| Loading | 예상 레이아웃을 유지하는 절제된 skeleton 또는 progress를 즉시 표시 |
| Empty | 무엇이 없는지 설명하고 가장 관련 있는 다음 행동 하나를 제공 |
| Error | 문제를 쉬운 문장으로 설명하고 가능한 경우 재시도·입력 복구 제공 |
| Success | 완료된 콘텐츠를 먼저 보여주고 중요한 저장만 간결하게 확인 |

자동으로 사라지는 메시지에 핵심 정보를 의존하지 않으며 screen reader가 상태 변화를 인지할 수 있게 한다.

## Safe area와 반응형

- 기준 뷰포트는 390px이며 page-level 가로 스크롤을 허용하지 않는다.
- 상단·하단 고정 UI는 `env(safe-area-inset-*)`를 반영한다.
- 768px 이상에서는 여백이나 열을 늘리되 카드 이미지와 핵심 행동의 우선순위를 바꾸지 않는다.
- fixed height보다 content-driven height, wrapping, fluid grid를 사용한다.

## 접근성

- semantic HTML과 논리적 heading 순서를 ARIA보다 먼저 사용한다.
- 키보드만으로 모든 행동에 접근하고 포커스 순서와 표시를 유지한다.
- body text는 4.5:1, 큰 굵은 텍스트는 3:1 이상을 목표로 실제 token 확정 시 측정한다.
- 확대 텍스트에서 수평 배치를 수직으로 전환하고 의미 있는 내용을 숨기지 않는다.
- motion은 상태 이해에 필요한 경우에만 짧게 사용하며 `prefers-reduced-motion`에서 제거·단순화한다.
- swipe와 drag 같은 gesture에는 항상 보이는 대체 행동을 제공한다.

## 피해야 할 디자인

- 공식 포켓몬 앱의 색, 로고, 캐릭터, 아이콘, 화면 구성을 직접 모방하는 것
- 과도한 그라데이션, glassmorphism, blur와 강한 그림자
- 장식 배지와 지표가 밀집된 대시보드
- 카드 이미지 위를 가리는 컨트롤과 메타데이터
- 의미 없는 bounce, parallax, 자동 재생 애니메이션
- placeholder-only 입력, 색상-only 상태, 작은 아이콘-only 행동
- 확정되지 않은 임시 색상·치수를 디자인 시스템의 최종값으로 취급하는 것
