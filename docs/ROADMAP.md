# Pokefolio 로드맵

로드맵은 구현 순서이며 일정 약속이 아니다. 각 Phase는 완료 조건을 검증한 뒤 다음 단계로 이동한다.

## Phase 0: 문서와 규칙

**목표:** 구현 전에 제품, 디자인, 데이터, API, 개발 기준을 합의한다.

**작업**

- 저장소 구조와 모노레포 경계 정의
- 제품 범위, 로드맵, 화면, ERD, API 초안 작성
- 코딩·테스트·에이전트 작업 규칙 작성
- 주요 기술 결정을 ADR로 기록

**완료 조건**

- 요구된 문서와 빈 디렉터리 유지 파일이 존재한다.
- 문서 링크와 기술 결정이 서로 충돌하지 않는다.
- 애플리케이션 코드와 dependency가 없다.
- 사용자 리뷰를 받아 Phase 1 범위를 확정한다.

## Phase 1: Frontend/Backend scaffold

**목표:** 최소 실행·검증 가능한 개발 기반을 만든다.

**작업**

- Next.js App Router, TypeScript strict, Tailwind CSS, shadcn/ui, npm 초기화
- FastAPI, SQLAlchemy 2.x, Alembic, Pydantic v2, pytest 초기화
- 공통 환경 설정, health API, API client 경계 구성
- lint, typecheck, test, build와 로컬 실행 명령 확정

**완료 조건**

- frontend와 backend가 독립적으로 실행된다.
- health 요청이 명시된 계약으로 응답한다.
- lint, typecheck, test, build 또는 해당 영역의 검증 명령이 통과한다.
- 앱 코드와 데이터베이스 파일이 Git ignore 정책을 따른다.

## Phase 2: Mock UI

**목표:** 실제 데이터 연결 전 승인된 모바일 경험을 검증한다.

**작업**

- 역할 기반 provisional 디자인 토큰과 기본 레이아웃 구성
- Home, Collection, Card Detail, Binder, Opening Log, Wishlist mock 화면 구현
- loading, empty, error, success mock 상태 구현
- 하단 내비게이션과 화면 이동 흐름 검증

**완료 조건**

- 390px에서 모든 핵심 화면이 가로 스크롤 없이 표시된다.
- 768px 이상에서도 정보 우선순위가 유지된다.
- 키보드, 포커스, 대비, 확대 텍스트, safe area, reduced motion 검수를 통과한다.
- 승인된 목업 방향과 화면 문서의 차이가 기록되고 승인된다.

## Phase 3: Collection

**목표:** 카드 검색과 보유 컬렉션 관리의 핵심 흐름을 완성한다.

**작업**

- Card, UserCard, WishlistItem 모델과 migration 구현
- 카드 조회·검색, 보유 카드, 위시리스트 API 구현
- Collection, Card Detail, Wishlist 화면을 API와 연결
- 수량, 상태, 메모, 중복 표시와 오류 복구 구현

**완료 조건**

- 카드 검색, 추가, 수정, 제거, 위시리스트 흐름이 동작한다.
- unique constraint와 validation이 중복·잘못된 입력을 차단한다.
- unit, API, component와 핵심 e2e 테스트가 통과한다.
- 빈 컬렉션과 네트워크 오류에서도 다음 행동을 안내한다.

## Phase 4: Binder

**목표:** 보유 카드를 가상 바인더 페이지에 배치하고 감상할 수 있게 한다.

**작업**

- Binder, BinderPage, BinderSlot 모델과 migration 구현
- 바인더·페이지·슬롯 API 구현
- 바인더 목록, 상세, 슬롯 선택·교체·비우기 구현
- 카드 이미지 비율과 3×3 페이지 레이아웃 검증

**완료 조건**

- 바인더 생성부터 슬롯 배치와 재조회까지 동작한다.
- 한 슬롯과 한 카드 배치의 제약 및 삭제 정책이 테스트된다.
- 390px에서 카드 이미지가 식별 가능하고 조작 영역이 접근 가능하다.
- 보유하지 않은 카드의 잘못된 배치를 서버가 거부한다.

## Phase 5: Opening Log

**목표:** 팩 개봉의 맥락과 획득 카드를 일기로 기록한다.

**작업**

- OpeningLog, OpeningLogCard 모델과 migration 구현
- 목록, 생성, 상세, 수정, 삭제 API 구현
- 개봉일기 입력, 획득 카드 선택, 목록과 상세 화면 구현
- 저장 실패, 중복 카드, 부분 입력 복구 흐름 설계

**완료 조건**

- 날짜·세트·메모와 획득 카드를 하나의 기록으로 저장한다.
- 저장은 transaction으로 원자성을 유지한다.
- 최근 기록과 상세가 일관된 데이터를 표시한다.
- unit, API, integration, 주요 e2e 테스트가 통과한다.

## Phase 6: Rule-based recommendation

**목표:** 설명 가능한 규칙으로 수집과 교환 후보 탐색을 돕는다.

**작업**

- 추천 입력 데이터와 사용자에게 보일 근거 정의
- 위시리스트, 중복, 세트·카테고리 선호 규칙 구현
- 추천 결과와 이유, 숨기기·피드백 흐름 구현
- 개인정보와 편향 위험 검토

**완료 조건**

- 모든 추천에 사람이 이해할 수 있는 근거가 표시된다.
- 추천 실패가 컬렉션 핵심 기능을 막지 않는다.
- 규칙별 deterministic 테스트와 경계 사례가 통과한다.
- 생성형 AI 없이도 최소 추천 가치가 검증된다.

## Phase 7: OCR/card recognition

**목표:** 카메라나 이미지에서 카드 후보를 찾아 등록을 보조한다.

**작업**

- 이미지 입력, 권한, 업로드·삭제와 개인정보 정책 정의
- OCR/인식 제공자와 비용·정확도 비교
- 복수 후보, 신뢰도, 수동 확인 UX 구현
- 실패·오인식·저조도 사례 평가

**완료 조건**

- 사용자가 인식 결과를 확인·수정한 뒤에만 컬렉션에 반영된다.
- 원본 이미지 보관 정책과 삭제 동작이 명확하다.
- 목표 데이터셋에서 합의한 정확도와 응답 시간 기준을 만족한다.
- 인식 기능 없이도 수동 등록 흐름이 유지된다.

## Phase 8: Trade

**목표:** 위시리스트와 중복 카드를 바탕으로 안전한 교환 준비 경험을 제공한다.

**작업**

- 다중 사용자, 인증, 공개 범위와 차단 정책 설계
- 교환 후보 매칭과 제안 상태 모델 정의
- 사용자 동의, 신고, 개인정보 노출 최소화 흐름 구현
- 거래 중재·결제·배송 포함 여부 결정

**완료 조건**

- 권한과 공개 범위가 API와 UI에서 일관되게 적용된다.
- 사용자가 공유할 카드와 정보를 명시적으로 선택한다.
- 악용, 스팸, 차단, 신고의 핵심 위험이 테스트된다.
- 법적·운영 책임과 지원 범위가 문서화된다.
